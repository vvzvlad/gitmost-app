# AGENTS.md

Руководство для агентов и разработчиков по проекту **Docmost** — нативному
macOS-приложению (Swift + AppKit + WebKit), которое показывает веб-интерфейс
Docmost для нескольких серверов во вкладках одного окна.

Этот файл адаптирует стандартное руководство по «новому проекту» к нативному
macOS GUI-приложению. Стандартное руководство написано под Python/Docker
серверные сервисы, поэтому значительная часть его машинерии (Docker, ghcr,
pydantic, `.env`, data-volume) здесь намеренно не применяется. Отклонения
перечислены ниже в разделе «Отклонения от стандартного руководства».

## Структура проекта

```
Sources/DocmostCore/      # Чистая логика (модели + персистентность), покрыта тестами
  Server.swift            #   модель сервера
  ServerStore.swift       #   хранилище серверов (UserDefaults) + нормализация URL
Sources/Docmost/          # macOS-приложение (AppKit/WebKit UI), тонкая надстройка над DocmostCore
  main.swift              #   тонкая точка входа (создаёт NSApplication и делегата)
  AppDelegate.swift
  MenuBuilder.swift
  Controllers/            #   MainWindowController, MainViewController, SettingsWindowController
  Views/                  #   TabBarView
  WebKit/                 #   WebTab (обёртка над WKWebView)
Tests/DocmostCoreTests/   # Юнит-тесты для DocmostCore (XCTest)
Resources/                # Ассеты приложения (иконки и т.п.), если есть
Makefile                  # Единая точка входа: сборка/упаковка Docmost.app и прочие действия
Package.swift             # Манифест Swift Package Manager (зависимости и таргеты)
```

Код разделён на две части: вся UI-независимая логика живёт в библиотеке
`DocmostCore` и обязательно покрывается тестами; приложение `Docmost` —
тонкий слой UI поверх неё. Бинарник исполняемого таргета называется `Docmost`,
именно его упаковывает цель `make build`.

## Окружение и установка

- Требуется macOS 14.0 или новее.
- Требуется тулчейн Swift: Xcode 15+ или Command Line Tools.
- Никакого virtualenv и pip — это не Python-проект. Зависимости управляются
  через Swift Package Manager (`Package.swift`).

## Команды

Все рутинные действия — через `make` (см. `make help`):

| Команда      | Назначение                                            |
|--------------|-------------------------------------------------------|
| `make build` | Релизная сборка `Docmost.app` (компиляция+упаковка)   |
| `make run`   | Собрать и запустить `Docmost.app`                     |
| `make test`  | Прогнать юнит-тесты (`swift test`)                    |
| `make debug` | Отладочная сборка SwiftPM (без `.app`)                |
| `make clean` | Удалить артефакты сборки                              |
| `make help`  | Показать список доступных целей                       |

## Конвенции

- Комментарии в коде — на английском.
- Строки пользовательского интерфейса — на английском.
- Никаких захардкоженных адресов серверов или секретов: серверы настраиваются
  в UI приложения во время работы и хранятся локально в `UserDefaults`.
- Тесты обязательны для новой логики в `DocmostCore`.
- CI блокирует сборку до прохождения тестов (см. `.github/workflows/ci.yml`).

## Отклонения от стандартного руководства

Причина всех отклонений: **это нативное macOS GUI-приложение, а не
Python/Docker-сервис.**

- **Нет Docker / `Dockerfile` / `.dockerignore` / docker-compose / ghcr.io /
  watchtower / Traefik** — macOS .app не может работать в Linux-контейнере и
  распространяется как подписанный бандл.
- **Нет `pydantic-settings` / `.env` / `.env.example`** — у приложения нет
  секретов или конфигурации времени развёртывания; список серверов вводится
  пользователем во время работы.
- **Нет каталога `data/` (volume)** — состояние времени выполнения хранится в
  `UserDefaults` (нативное хранилище macOS), а не в смонтированном томе.
- **Нет `requirements.txt`** — зависимости управляются SwiftPM (`Package.swift`).
